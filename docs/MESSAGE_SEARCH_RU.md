# Поиск сообщений OfficeChat

## Возможности v0.1

OfficeChat ищет сообщения в доступных пользователю группах, личных диалогах и обсуждениях. Поиск охватывает текущий текст сообщения, безопасное исходное имя вложения, username и отображаемое имя отправителя. Удалённые и архивные сообщения из поиска исключаются.

Панель поиска открывается кнопкой рядом с поиском сайдбара или сочетанием `Ctrl+K` / `Cmd+K`. Доступны глобальная и текущая область чата, отправитель, период и фильтр наличия вложений. Запрос отправляется после debounce 350 мс; устаревший HTTP-запрос отменяется через `AbortController`. Поисковые запросы и результаты не сохраняются в `localStorage` или Valkey.

## PostgreSQL

Миграция `20260704_0018` добавляет GIN-индексы выражения `to_tsvector('simple', coalesce(..., ''))` для body трёх таблиц сообщений и `original_filename` трёх таблиц вложений. Конфигурация `simple` выбрана для смешанного RU/EN-текста, username, IP-адресов, имён файлов и технических терминов без языкового stemming.

Ранжирование учитывает совпадение фразы в текущем body, полнотекстовую релевантность, отправителя и имя вложения. Стабильный порядок и cursor pagination используют `rank`, `created_at`, затем `message.id`. Лимит по умолчанию — 30, максимум — 100; длина нормализованного запроса — от 2 до 200 символов.

В v0.1 нет внешнего поискового сервиса и `pg_trgm`. Для очень крупных установок следует анализировать реальные запросы через `EXPLAIN (ANALYZE, BUFFERS)` и отдельно проектировать шардирование или OpenSearch.

## Доступ и приватность

- Group search требует актуального членства в группе.
- Direct search доступен только двум участникам conversation; admin-роль не даёт доступа к чужой переписке.
- Discussion search требует актуального членства в discussion.
- Проверки выполняются на backend для каждого типа чата; удаление membership немедленно исключает данные из новых запросов.
- Snippet формируется только после SQL-фильтра доступа, возвращается как ограниченный plain text и не содержит HTML.
- `storage_path`, stored filename, токены и служебные секреты не индексируются и не возвращаются.
- Физически удалённый retention-файл не раскрывает путь. Сохранённое безопасное `original_filename` остаётся частью метаданных сообщения и может участвовать в поиске.

Обычный поиск не создаёт Audit Log events. Параметр `q` редактируется в access logs как `q=[REDACTED]`; сырой текст запроса не журналируется на INFO.

## API

```text
GET /api/search/messages
GET /api/search/context
```

`GET /api/search/messages` принимает `q`, необязательные `chat_type`, `chat_id`, `sender_id`, `date_from`, `date_to`, `has_attachment`, `limit` и `cursor`.

`GET /api/search/context` принимает `chat_type`, `chat_id`, `message_id`, `before` и `after`. Endpoint повторно проверяет доступ и принадлежность target к чату, возвращает target и соседние сообщения в обычном порядке. Используются существующие публичные схемы, поэтому сохраняются replies, reactions, mentions, attachments и поля закрепления `is_pinned`, `pin_id`, `pinned_at`. Переход из pinned strip использует тот же механизм jump-to-message.

## Переход и deep links

Внутренняя ссылка имеет вид:

```text
/ru/app?chatType=direct&chatId=<uuid>&messageId=<uuid>
```

После перехода OfficeChat выбирает чат, загружает контекст, центрирует target и подсвечивает его примерно на 3 секунды. Та же ссылка для пользователя без доступа возвращает нейтральную ошибку без метаданных сообщения.

Во время просмотра исторического контекста новые WebSocket-события не заменяют загруженную историю и не прокручивают её вниз. Кнопка «Перейти к последним» возвращает обычный поток.

## Unread state

Загрузка search context сама по себе не вызывает `POST /api/read-state`. Пока открыт исторический контекст, автоматический visible-chat marker отключён, поэтому переход к старому сообщению не очищает более новые unread-сообщения. После возврата к последним сообщениям работает обычная visibility/debounce-логика.

## Ограничения и roadmap

Не реализованы: расширенный синтаксис и точные фразы, сохранённые запросы, история поиска, OCR и поиск внутри документов, отдельные eDiscovery-права, `pg_trgm` и OpenSearch. Эти функции требуют отдельной модели разрешений и эксплуатации.
