# Гранулярные права OfficeChat

Granular Permissions v0.1 добавляет слой специальных прав поверх существующих ролей. Роли продолжают управлять широким доступом к приложению, а специальные права предназначены для чувствительных возможностей, которые нельзя автоматически выдавать каждому администратору или модератору.

## Чем права отличаются от ролей

- `role` определяет базовую область доступа: обычный пользователь, администратор, bot и так далее.
- `permissions` определяют отдельные чувствительные действия.
- Frontend использует права только для показа или скрытия UI.
- Backend остаётся авторитетным источником и обязан проверять права в каждом чувствительном endpoint.

В v0.1 каталог содержит:

- `can_broadcast` - будущая отправка массовых или targeted announcements.
- `can_pin_messages` - будущая возможность закреплять и откреплять сообщения в доступных чатах.

Массовые рассылки и закреплённые сообщения в этой задаче не реализованы.

## Эффективные права

- `superadmin` имеет все активные специальные права неявно.
- Для `superadmin` не создаются лишние строки в `user_permissions`.
- `admin`, `moderator`, `group_owner` и обычные пользователи не получают `can_broadcast` или `can_pin_messages` автоматически.
- Отключённый пользователь (`is_active=false`) не имеет эффективных специальных прав.
- Bot accounts не получают специальные права в v0.1.
- `GET /api/auth/me` возвращает `permissions` для UI, но JWT не хранит авторитетный список прав.

Такой подход позволяет отзывать право без ожидания истечения JWT.

## Управление

Назначать и отзывать специальные права может только `superadmin`.

API:

```text
GET /api/admin/permissions
GET /api/admin/users/{user_id}/permissions
PUT /api/admin/users/{user_id}/permissions
```

Тело bulk update:

```json
{
  "permissions": ["can_broadcast", "can_pin_messages"]
}
```

Endpoint валидирует ключи, обновляет grants атомарно, не создаёт дубликаты и не пишет audit events для неизменённых прав. Нельзя назначать права самому себе, bot account или управлять правами без роли `superadmin`.

## UI

В Admin User Management (`/ru/admin/users`) у `superadmin` в drawer пользователя отображается раздел "Специальные права".

- `superadmin` видит свои права как включённые ролью, без возможности снять implicit grants.
- Обычный `admin` не видит элементы управления специальными правами.
- Для `can_broadcast` показано предупреждение о высоком уровне доступа.
- Пользовательский UI должен считать эти права подсказкой для видимости, а не защитой.

## Audit

Изменения прав пишутся в `audit_events` в той же транзакции, что и изменение grant:

- `permission.granted`
- `permission.revoked`

Категория: `security`.

В деталях сохраняется только безопасное значение:

```json
{
  "permission": "can_broadcast"
}
```

JWT, пароли, токены и session data не сохраняются.

## Обновление открытых сессий

После изменения прав backend отправляет affected user событие через персональный канал:

```json
{
  "type": "permissions.updated",
  "permissions": ["can_pin_messages"]
}
```

Frontend обновляет видимые controls без принудительного logout. Backend всё равно проверяет права по базе данных.

## Контракт для будущих функций

Будущие endpoints рассылок должны использовать:

```python
require_permission(CAN_BROADCAST)
```

Будущие endpoints закрепления сообщений должны использовать:

```python
require_permission(CAN_PIN_MESSAGES)
```

Скрытой кнопки во frontend недостаточно. Каждая чувствительная backend-мутация должна выполнять серверную проверку.

## Ограничения v0.1

- Нет UI или API массовой рассылки.
- Нет закрепления сообщений.
- Нет делегированного администрирования прав.
- Нет отдельного permission cache; indexed PostgreSQL lookup является источником истины.
- Multi-instance доставка `permissions.updated` позже должна использовать Valkey pub/sub или другой broker.
